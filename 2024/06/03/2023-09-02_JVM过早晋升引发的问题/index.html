JVM过早晋升引发的问题

这篇文章主要是讲解了JVM过早晋升引发的问题

前言

这种场景主要是出现在分代垃圾收集器上面，专业的术语称为“Premature Promotion”。

在JVM中90%的对象都是朝生夕死的，只有在Old区经过几次GC的洗礼才会晋升到老年代，每经历一次minor GC，没有被回收的对象的Age则加1，最大通过 -XX:MaxTenuringThreshold 来控制。

过早晋升会导致浮动垃圾、大对象担保失败、GC停顿时间长等等问题，但是这些现象通常不是立刻发生，而是经过一定时间的GC才会暴漏这些问题。

判断发生过早晋升的现象

分配速率接近于晋升速度，对象晋升年龄较小

通过GC日志查看堆内存的分配和释放情况。通过计算不同GC事件之间的时间差和内存变化，可以得到内存分配速率。

在GC日志中，关注Minor GC（Young GC）和Major GC（Old GC）事件。每次Minor GC后，如果有对象晋升到老年代，日志中会有相应的记录。

通过日志中的时间戳和内存使用情况，计算对象晋升的频率和数量。

GC日志中观察new threshold

GC 日志中出现“Desired survivor size 107347968 bytes, **new threshold 1(max 6)**”等信息，说明此时经历过一次 GC 就会放到 Old 区。

Full GC频繁，且经历一次GC后比例变化非常大

比如说 Old 区触发的回收阈值是 80%，经历过一次 GC 之后下降到了 10%，这就说明 Old 区的 70% 的对象存活时间其实很短，如下图所示，Old 区大小每次 GC 后从 2.1G 回收到 300M，也就是说回收掉了 1.8G 的垃圾，只有 300M 的活跃对象。整个 Heap 目前是 4G，活跃对象只占了不到十分之一。

过早晋升的原因

主要的原因分为以下两点：

  • Young/Eden 区过小： Young/Eden区过小的直接后果就是 Eden 被装满的时间变短，本应该在Young区回收的对象参与多次minor GC 并晋升到Old，Young GC 采用的是复制算法，我们知道 copying 耗时远大于 mark，也就是 Young GC 耗时本质上就是 copy 的时间（CMS 扫描 Card Table 或 G1 扫描 Remember Set 出问题的情况另说），没来及回收的对象增大了回收的代价，所以 Young GC 时间增加，同时又无法快速释放空间，Young GC 次数也跟着增加。

无法GC掉对象会带来一个问题，引发动态年龄计算，JVM通过-XX:MaxTenuringThreshold 参数来控制晋升年龄，每经过一次 GC，年龄就会加一，达到最大年龄就可以进入 Old 区，最大值为 15（因为 JVM 中使用 4 个比特来表示对象的年龄）。设定固定的 MaxTenuringThreshold 值作为晋升条件的缺点：

  • MaxTenuringThreshold 如果设置得过大，原本应该晋升的对象一直停留在 Survivor 区，直到 Survivor 区溢出，一旦溢出发生，Eden + Survivor 中对象将不再依据年龄全部提升到 Old 区，这样对象老化的机制就失效了。
  • MaxTenuringThreshold 如果设置得过小，过早晋升即对象不能在 Young 区充分被回收，大量短期对象被晋升到 Old 区，Old 区空间迅速增长，引起频繁的 Major GC，分代回收失去了意义，严重影响 GC 性能。

相同的应用在不同时间段的表现不同，特殊任务的执行或者流量的变化，都会导致对象的生命周期分布发生波动，如果设置固定的阈值，无法动态的适应变化的话，则会造成以上的问题，所以Hotspot会使用动态年龄计算的方式来调整晋升的阈值。

具体动态计算可以看一下 Hotspot 源码，具体在 /src/hotspot/share/gc/shared/ageTable.cpp 的 compute_tenuring_threshold 方法中：

可以看到Hotspot遍历所有的对象，从所有年龄为0的对象的占用空间进行累加，如果加上年龄n的所有对象空间后，大于预期的占用量后，则结束这个循环，若 n 小，则阈值为 n，若 n 大，则只能去设置最大阈值为 MaxTenuringThreshold。，在对象过早晋升的场景下，动态年龄计算出发后会导致更多的对象进入Old区，造成资源浪费（因为年龄小的偏多，更容易达到阈值）。

如何解决？

知道问题的根本原因后我们就有解决的方向，如果是 Young/Eden 区过小，我们可以在总的 Heap 内存不变的情况下适当增大 Young 区，具体怎么增加？一般情况下 Old 的大小应当为活跃对象的 2~3 倍左右，考虑到浮动垃圾问题最好在 3 倍左右，剩下的都可以分给 Young 区。（-XX:NewRatio`默认通常为2，也就是新生代：老年代=1：2）

就拿一次典型案例来说，原配置为Young 1.2G + Old 2.8G，观察CMS GC的情况发现存活对象大概在300~400M左右，于是调整Old为1.5GB左右，剩下的都给Young区。仅仅调了一个Young区大小的参数后（-Xmn），整个JVM一分钟Young GC从26次降低到了11次。单词时间也没有增加，总的时间从1100ms降低到500ms，CMS GC次数也从40分钟一次降低到了7小时30分钟一次。

如果是分配速率过大：

  • 偶发较大：通过内存分析工具找到问题代码，从业务逻辑上做一些优化。
  • 一直较大：当前的 Collector 已经不满足 Mutator 的期望了，这种情况要么扩容 Mutator 的 VM，要么调整 GC 收集器类型或加大空间。
